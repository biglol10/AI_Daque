## Executive Summary

AI 셀프 다큐멘터리는 AI 기술을 활용하여 개인의 삶의 이야기를 **귀여운 AI 카툰 캐릭터 + 시대배경 장면**으로 구성된 1~2분 다큐멘터리 영상으로 자동 제작하는 플랫폼이다. 사용자는 AI 인터뷰 또는 템플릿 작성으로 인생 스토리를 입력하고, 자기 얼굴 사진을 업로드하면 AI가 이를 귀여운 카툰 캐릭터로 변환한다. 인생 시기별(10대, 20대, 30대...) 시대정보를 자동 수집하여 해당 시대의 배경을 AI로 생성하고, 캐릭터가 그 배경 안에서 살아가는 다큐멘터리 영상을 제작한다. 사용자 음성 클로닝 또는 샘플 보이스로 나레이션이 입혀진다.

| 관점 | 내용 |
|------|------|
| **Problem** | 개인의 소중한 삶의 이야기를 전문적 다큐멘터리로 기록하고 싶지만, 전문 제작팀 없이는 불가능하며 기존 AI 도구는 개인 서사에 특화되지 않음 |
| **Solution** | AI 인터뷰/템플릿으로 인생 스토리 수집 → 얼굴 기반 귀여운 카툰 캐릭터 생성 → 시대배경 AI 생성 → 보이스 클로닝 나레이션이 결합된 1~2분 다큐멘터리 영상 자동 제작 |
| **Function/UX Effect** | 인터뷰/작성 → 캐릭터+배경 생성 → 다큐 영상 1-click 생성의 3단계 플로우. 에피소드 시리즈로 인생 시기별 영상 시리즈 제작 가능 |
| **Core Value** | "모든 사람의 삶에는 다큐멘터리로 남길 가치가 있다" - 나를 닮은 귀여운 캐릭터가 내 인생을 살아가는 영상으로 개인 다큐멘터리 민주화 |

# Part 2: Strategy Analysis (pm-strategy)

## 2. Value Proposition (JTBD 6-Part Framework)

### Jobs-to-be-Done Statement

> **When** [부모님이 나이 들어가시거나, 인생의 중요한 전환점을 맞이할 때],
> **I want to** [소중한 삶의 이야기를 전문적이고 감동적인 영상으로 기록하고 싶지만],
> **So I can** [가족과 후세대에게 의미 있는 유산을 남기고, 삶의 가치를 재확인하고 싶다].

### 6-Part Value Proposition

| Part | 내용 |
|------|------|
| **1. Target Customer** | 30-60대, 가족 이야기를 보존하고 싶은 사람. 부모님/조부모님의 이야기를 기록하려는 자녀. 인생 전환점(은퇴, 결혼기념일)을 맞은 사람 |

| **6. Alternatives** | (1) 전문 다큐 제작업체 의뢰 (비용 500만원+) (2) StoryWorth/Legacium (텍스트 회고록만) (3) 직접 유튜브 편집 (높은 기술 장벽) (4) Mootion (일반 다큐, 개인 서사 미지원) |

## 4. SWOT Analysis

| | Helpful | Harmful |
|---|---------|---------|
| **Internal** | **Strengths**: (1) 니치 시장 선점 기회 (개인 서사 + 다큐 결합) (2) 한국어 네이티브 특화 가능 (3) AI 기술 접근성 높음 (API 기반) (4) BaaS 기반 빠른 MVP 개발 | **Weaknesses**: (1) AI 영상 품질 외부 API 의존 (2) 초기 브랜드 인지도 없음 (3) 개인 데이터 신뢰 확보 필요 (4) 단독 개발자/소규모 팀 리소스 한계 |
| **External** | **Opportunities**: (1) 고령화 사회 -> 기록 보존 수요 증가 (2) AI 영상 기술 급속 발전 (3) 한국 시장 경쟁자 부재 (4) 효도/가족 문화와 시너지 (5) 숏폼 콘텐츠 트렌드 | **Threats**: (1) 글로벌 빅테크 AI 영상 도구 진입 (2) 개인정보 보호 규제 강화 (3) AI 생성 콘텐츠에 대한 사회적 거부감 (4) 기존 영상 편집 도구의 AI 기능 강화 |

### SO 전략 (Strengths x Opportunities)
- 한국어 특화 + 고령화 사회 수요를 결합하여 "효도 다큐" 시장 선점
- BaaS 기반 빠른 개발로 한국 시장 경쟁자 부재 시 선점 효과 극대화

### WT 전략 (Weaknesses x Threats)
- 프라이버시 정책을 최우선으로 설계하여 신뢰 기반 구축
- 외부 AI API 의존도를 분산하여 특정 서비스 종속 방지
- 커뮤니티 기반 성장으로 마케팅 비용 최소화

## 12. Growth Loops

### Loop 1: "감동 공유" 바이럴 루프
```
사용자가 다큐 완성 -> 가족/SNS에 공유 -> 시청자 감동
-> "나도 만들고 싶다" -> 신규 가입 -> 다큐 제작 -> 공유...
```
- **Key Metric**: 다큐 공유율, 공유 -> 가입 전환율
- **강화 전략**: 공유 시 워터마크에 서비스 브랜딩, 쉬운 공유 버튼

### Loop 2: "가족 초대" 확장 루프
```
자녀가 부모 인터뷰 -> 완성 다큐를 형제/자매에게 공유
-> 형제가 "나도 부모님 다큐 만들자" -> 추가 프로젝트
-> 또는 "내 다큐도 만들자" -> 셀프 다큐 전환
```
- **Key Metric**: 가족 초대율, 사용자당 프로젝트 수

### Loop 3: "콘텐츠 품질" 데이터 루프
```
사용자 인터뷰 데이터 축적 -> AI 서사 구조화 품질 개선
-> 더 좋은 다큐 생성 -> 높은 만족도 -> 더 많은 사용자
-> 더 많은 데이터 -> AI 품질 향상...
```
- **Key Metric**: AI 서사 품질 점수, 사용자 만족도 추이

## 14. Technical Architecture (High-Level)

### 14.1 기술 스택

> **아키텍처 결정: 하이브리드 (TypeScript 프론트 + Python AI 백엔드)**
>
> - AI 생태계(LangGraph, diffusers, HuggingFace)가 Python에 압도적으로 집중되어 있음
> - 인터뷰 엔진의 기술 moat인 LangGraph는 Python이 네이티브 (JS 버전 생태계 1/5 수준)
> - 프론트엔드는 개발자의 기존 역량(FE/BE)을 살려 Next.js/TypeScript 유지
> - 통신: Next.js API Routes → FastAPI REST 호출

### 14.2 시스템 아키텍처

> **핵심 결정: LangGraph(Agentic) + Pipeline(DAG) 하이브리드**
>
> - **인터뷰 엔진**: 비정형, 판단 필요 → LangGraph StateGraph (조건부 분기, 체크포인터)
> - **생성 파이프라인**: 정형, 순서 확정 → 단순 DAG (`asyncio.gather`로 병렬)
> - Agent로 만들면 오버엔지니어링인 부분은 파이프라인으로, 판단이 필요한 부분만 Agentic으로

```
│  └────┬─────┘ └────┬─────┘ └────┬─────┘ └──────┬────────────┘  │

        ▼            ▼            ▼               ▼
┌─────────────────────────────────────────────────────────────────┐

│              → FastAPI AI Backend REST 호출                      │
└───────────────────────────┬─────────────────────────────────────┘
                            │
          ┌─────────────────┼─────────────────┐
          ▼                                   ▼
┌─────────────────────────┐   ┌───────────────────────────────────┐
│  LangGraph (Agentic)    │   │  Generation Pipeline (DAG)        │
│  ═══════════════════    │   │  ═══════════════════════          │
│                         │   │                                   │
│  [질문생성] → [답변수신] │   │  서사확정 → [asyncio.gather]      │
│      ↕                  │   │              ├─ 캐릭터 생성       │
```

### 14.3 AI 가드레일 시스템

> 개인의 인생 이야기 + 얼굴 + 음성을 다루는 서비스 특성상, AI 가드레일은 선택이 아닌 필수.
> LangGraph 노드로 자연스럽게 파이프라인에 삽입.

| 영역 | 위험 | 가드레일 구현 | 위치 |
|------|------|-------------|------|
| **인터뷰 콘텐츠** | 트라우마/학대/자해 등 민감 내용 표출 | LLM Classifier로 민감 주제 감지 → 부드러운 전환 응답 ("이 부분은 힘드셨겠네요. 다른 이야기로 넘어갈까요?") | LangGraph 노드 |
| **캐릭터 생성** | 부적절한 이미지, 인종/성별 왜곡 | Negative prompt 강제 적용 + NSFW safety classifier 통과 검증 | 생성 파이프라인 |
| **보이스 클로닝** | 타인 음성 도용, 딥페이크 악용 | 동의 확인 UI 필수 + "본인 음성임을 확인합니다" 체크 + 생성 음성 워터마크 | 프론트엔드 + 파이프라인 |
| **시대정보** | 역사적 사실 오류, 민감 정치 이슈 | 한국 시대 DB 교차 검증 + 정치/종교 중립화 필터 | LangGraph RAG 노드 |
| **개인정보(PII)** | 인터뷰에서 타인 실명/주소/전화번호 노출 | PII 감지 필터 → 자동 마스킹 제안 | LangGraph 노드 |
| **비용 보호** | 무한 재생성으로 API 비용 폭증 | 사용자당 일일 생성 횟수 제한 + 단계별 렌더링(저해상 미리보기→확정→고해상) | API Gateway |

```

  질문생성 → 답변수신 → [가드레일: 민감 체크] ─── 정상 ──→ 분기판단 → 다음질문
                                    │

                                    │
```

### 15.2 Top 5 Risks & Mitigation

**Risk 1: AI 캐릭터 유사성 (Score: Critical)**
- Mitigation: 복수 스타일 프리셋 제공, 사용자 피드백 기반 재생성, 얼굴 특징(눈/코/입 비율) 보존 알고리즘 검증 PoC
- Monitoring: 캐릭터 재생성 횟수, "내 캐릭터 만족도" 설문

**Risk 2: 다중 AI 파이프라인 비용 (Score: Critical)**
- Mitigation: 단계별 렌더링(스토리보드 미리보기=저해상→확정 후 고해상), 캐릭터/배경 캐싱, 비용 모니터링 대시보드
- Monitoring: 건당 생성 비용, API 호출 빈도, 사용자당 월 비용

**Risk 3: 개인정보 신뢰 (Score: Critical)**
- Mitigation: 프라이버시 바이 디자인, 얼굴/음성 데이터 "생성 후 삭제" 옵션, 데이터 보존 기간 명시, 국내 서버
- Monitoring: 가입 전환율, 얼굴 업로드 완료율, 이탈 지점 분석

**Risk 4: 시대배경 정확성 (Score: High)**
- Mitigation: 한국 시대 이벤트 DB 전문가 검수, LLM+DB 하이브리드 검증, 사용자 피드백 수정 루프
- Monitoring: 시대배경 관련 사용자 리포트 수, 수정 요청 빈도

**Risk 5: 사용자 완료율 (Score: High)**
- Mitigation: AI 인터뷰+템플릿 듀얼 입력, 마이크로 세션(5분 단위), 명대사 카드로 중간 성과물 제공
- Monitoring: 인터뷰 완료율, 템플릿 완성율, 다큐 생성 도달율

## Appendix B: 아키텍처 결정 기록 (ADR)

### ADR-001: LangGraph(Agentic) + Pipeline(DAG) 하이브리드

- **결정**: 인터뷰 엔진은 LangGraph, 생성 파이프라인은 단순 DAG
- **근거**: 인터뷰는 답변 기반 분기 판단이 필요(Agentic), 생성은 입력 확정 후 실행만(Pipeline). Agent로 생성을 만들면 오버엔지니어링
- **대안**: (1) 전체 LangGraph — 불필요한 복잡도 (2) 전체 파이프라인 — 인터뷰 분기 표현 불가

### ADR-002: Python + TypeScript 하이브리드 (FastAPI + Next.js)

- **결정**: 프론트엔드 Next.js(TS), AI 백엔드 FastAPI(Python), 2개 서비스 운영
- **근거**: AI 생태계(LangGraph, diffusers, HuggingFace)가 Python에 집중. LangGraph Python이 네이티브(JS 생태계 1/5 수준). 개발자 학습 가치 극대화
- **대안**: (1) 올 TypeScript — 배포 단순하나 LangGraph/AI 생태계 제약 (2) 올 Python — 프론트엔드 역량 낭비
- **트레이드오프**: 2개 서비스 운영 복잡도 증가, 하지만 스케일링 분리 가능

### ADR-003: AI 가드레일 필수 적용

- **결정**: 인터뷰 콘텐츠, 이미지 생성, 보이스 클로닝, 시대정보, PII 5개 영역에 가드레일 적용
- **근거**: 개인 인생사/얼굴/음성을 다루는 서비스 특성상 민감 콘텐츠/프라이버시 리스크가 높음. 출시 후 사후 대응은 신뢰 회복 불가
- **구현**: LangGraph 노드로 파이프라인에 자연스럽게 삽입 (별도 시스템 아님)
